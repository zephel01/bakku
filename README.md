# bakku

`bakku` is a cross-platform, encrypted, deduplicating backup CLI written in Go.
It stores backups in a content-addressable, restic-like repository: data is
split into content-defined chunks (keyed FastCDC), deduplicated by BLAKE3
content hash, zstd-compressed, and AES-256-GCM encrypted before being packed and
uploaded to a storage backend.

## Features

- **Content-addressable storage (CAS)** — all data is keyed by content hash;
  identical data is stored exactly once, across files and across snapshots.
- **Keyed FastCDC chunking** — content-defined chunking with a per-repository
  key (`github.com/PlakarKorp/go-cdc-chunkers`), so chunk boundaries are stable
  under small edits (good dedup) without being predictable from repository
  contents alone.
- **BLAKE3 hashing** — fast, parallel content hashing used both for
  chunk/blob addressing and dedup lookups.
- **AES-256-GCM encryption** — every data, tree, index, and snapshot blob is
  authenticated-encrypted with a subkey derived from a random master key,
  itself wrapped with an argon2id-derived key from your repository password.
- **zstd compression** — chunks are compressed before encryption to reduce
  storage and transfer size.
- **Deduplication** — new backups only upload chunks not already present in
  the repository; unchanged files across runs cost near-zero additional
  storage or bandwidth.
- Six storage backends (local disk, S3, SFTP, Google Drive, Dropbox, SMB),
  OS metadata capture (ownership/xattrs/hard links), GFS retention
  (`forget`/`prune`), integrity checking (`check`/`verify-restore`),
  OS-native job scheduling (`schedule`), and webhook notifications
  (`notify`). See the sections below for details.

Remaining gaps are listed in
[Not yet implemented](#not-yet-implemented-later-phases).

## Install

### Release binaries

Prebuilt binaries for darwin/linux/windows (amd64 + arm64) are attached to
each [GitHub Release](../../releases) tagged `vX.Y.Z`, generated by
`scripts/build-release.sh` (see [CI/CD](#cicd)). Download the archive for your
platform, extract it, and put `bakku` (or `bakku.exe`) on your `PATH`.

### go install

```sh
go install github.com/zephel01/bakku/cmd/bakku@latest
```

This builds from source and does not embed version metadata (`bakku
version` will print `dev`); use a release binary if you want a
self-reporting version string.

### Build from source

```sh
git clone https://github.com/zephel01/bakku
cd bakku
go build -o bakku ./cmd/bakku
```

Requires Go 1.26+.

## Concepts

- **Repository** — an encrypted, deduplicating store addressed by a URL or path.
- **Destination (dest)** — a friendly name mapped to a repository URL in the
  config file, so you can type `--repo laptop-backup` instead of the full URL.
- **Snapshot** — one backup run: a timestamped root tree plus metadata
  (hostname, paths, tags).

The repository password protects a random master key (wrapped with an
argon2id-derived key). All data, tree, index, and snapshot blobs are encrypted
with subkeys derived from the master key.

## Password

The password is resolved in this order:

1. `BAKKU_PASSWORD` environment variable
2. `--password-file <path>` (first line)
3. interactive prompt

## Quick start

```sh
# Create a repository on local disk (or NAS mount).
export BAKKU_PASSWORD='correct horse battery staple'
bakku init --repo file:///mnt/backups/laptop

# Register it as a named destination so you can refer to it by name.
bakku dest add laptop file:///mnt/backups/laptop

# Back up some paths (incremental: unchanged chunks are skipped).
bakku backup ~/Documents ~/Pictures --repo laptop --tag daily

# List snapshots.
bakku snapshots --repo laptop

# List the files in a snapshot (ID may be abbreviated).
bakku ls a1b2c3d4 --repo laptop

# Restore a snapshot into a directory.
bakku restore a1b2c3d4 --repo laptop --target /tmp/restore
```

## Commands

| Command | Description |
|---|---|
| `bakku init --repo <dest\|URL>` | Create a new repository. |
| `bakku backup <paths...> --repo <dest\|URL> [--tag t] [--exclude glob] [--parallel n] [--no-notify] [--use-vss]` | Incremental backup; prints new/reused chunk stats. |
| `bakku snapshots --repo <dest\|URL>` | List snapshots (short ID, time, host, tags, paths). |
| `bakku restore <snapID> --repo <dest\|URL> --target <dir> [--include glob] [--chown] [--restore-quarantine]` | Restore a snapshot. |
| `bakku ls <snapID> --repo <dest\|URL>` | List files in a snapshot. |
| `bakku diff <snapID1> <snapID2> --repo <dest\|URL>` | Show added/removed/changed paths between two snapshots. |
| `bakku forget --keep-last/--keep-daily/--keep-weekly/--keep-monthly/--keep-yearly N [--keep-tag t] [--dry-run] [--prune] --repo <dest\|URL>` | Apply a GFS retention policy and delete snapshots that fall outside it. |
| `bakku prune [--dry-run] [--no-notify] --repo <dest\|URL>` | Reclaim space: delete fully-unused packs, repack partially-used ones. |
| `bakku check [--read-data] --repo <dest\|URL>` | Verify repository structural integrity (index/pack/tree consistency); with `--read-data`, also re-hashes every blob. |
| `bakku verify-restore <snapID> [--sample pct] --repo <dest\|URL>` | Restore a random sample of a snapshot's files to a temp dir and verify content hashes. |
| `bakku dest add <name> <url>` / `dest list` / `dest remove <name>` | Manage destinations in the config file. |
| `bakku schedule install/uninstall/status` | Manage OS-native scheduled jobs. See [Scheduling](#scheduling). |
| `bakku version` / `bakku --version` | Print the version, commit, and build date. |

Global flags: `--repo`, `--config`, `--password-file`, `--json`
(`--json` produces machine-readable output for most commands: `backup`,
`snapshots`, `forget`, `prune`, `check`, `diff`, `verify-restore`).

`backup` and `prune` additionally accept `--no-notify` to skip the configured
webhook notification for that one run (see [Notifications](#notifications)).
`backup` also accepts `--use-vss` (Windows only; currently a stub that warns
and continues without a volume snapshot — see
[Not yet implemented](#not-yet-implemented-later-phases)).

## OS metadata (ownership, extended attributes, hard links)

On Linux and macOS, `backup` captures each file's owner uid/gid and extended
attributes, and detects hard links (same device+inode) so multiple names for
one file are stored once per snapshot instead of once per name.

- **Ownership**: captured always; restored only with `restore --chown`, and
  only takes effect when bakku is run as root/Administrator (silently
  skipped otherwise — this is expected, not an error). Failures during
  restore are reported as warnings, never abort the restore.
- **Extended attributes**: captured and restored automatically. On macOS, the
  Gatekeeper quarantine attribute (`com.apple.quarantine`) is excluded from
  restore by default so restored applications/documents are not re-blocked
  from opening; pass `restore --restore-quarantine` to restore it anyway.
- **Hard links**: detected via (device, inode) during `backup`; the first
  name seen stores real content, subsequent names sharing the same inode are
  stored as a lightweight reference and recreated with `os.Link` on
  `restore`, so restored files still share an inode and disk usage is not
  duplicated.
- **Windows**: ownership/xattrs have no direct equivalent and are not
  captured; long paths (>260 chars) are supported internally via the
  `\\?\` path prefix, and files are opened with `FILE_SHARE_DELETE` so
  files concurrently renamed/deleted by other processes do not abort a
  backup. Volume Shadow Copy (`--use-vss`) is not implemented — see
  [Not yet implemented](#not-yet-implemented-later-phases).
- **Other platforms**: all of the above are no-ops; backups/restores work
  normally, just without this extra metadata.

## Storage backends

All remote backends retry transient failures with exponential backoff
(200ms, 400ms, 800ms, plus jitter — 3 attempts total) and respect context
cancellation/timeouts. "Not found" style errors are not retried and satisfy
`errors.Is(err, os.ErrNotExist)`.

### S3 (and S3-compatible stores)

```
s3://bucket/prefix?endpoint=https://minio.example.com&region=us-east-1
```

- `bucket` — required. `prefix` — optional key prefix within the bucket.
- `endpoint` — optional; set this to point at a non-AWS S3-compatible
  service (MinIO, Backblaze B2, etc.). When set, the client uses
  path-style addressing (`UsePathStyle=true`).
- `region` — optional; forwarded to the AWS config loader.
- **Credentials**: resolved via the standard AWS SDK v2 credential chain
  (environment variables `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`,
  shared `~/.aws/credentials` and `~/.aws/config` files, EC2/ECS instance
  metadata, etc.). No bakku-specific credential flags exist.
- Uploads/downloads use `PutObject`/`GetObject` with HTTP `Range` for partial
  reads, and `ListObjectsV2` (paginated) for listing.

### SFTP

```
sftp://user@host:port/path
```

- `port` defaults to `22`; `user` defaults to the current OS user if omitted.
- **Authentication**, tried in order:
  1. `ssh-agent`, if `SSH_AUTH_SOCK` is set.
  2. Private keys `~/.ssh/id_ed25519` and `~/.ssh/id_rsa`.
  3. Password from the `BAKKU_SFTP_PASSWORD` environment variable.
- **Host key verification**: checked against `~/.ssh/known_hosts`. Set
  `BAKKU_SSH_INSECURE=1` to skip verification entirely (not recommended
  outside throwaway/test hosts).
- Saves are written to a temporary remote name and renamed into place for a
  pseudo-atomic upload.

### Google Drive

```
gdrive://folder-path
```

- `folder-path` is a `/`-separated path of Drive folder names (created on
  demand) under "My Drive"; repository keys map onto this folder hierarchy
  (e.g. key `data/ab/cd` becomes a file named `cd` inside
  `folder-path/data/ab`). Folder IDs are cached in memory per backend
  instance to avoid repeated lookups.
- **Setup**:
  1. Create an OAuth client ID (Desktop app) in Google Cloud Console and
     download the client-secret JSON.
  2. Set `BAKKU_GDRIVE_CREDENTIALS=/path/to/client_secret.json`.
  3. On first run, bakku prints an authorization URL; open it, sign in, and
     paste the resulting code back into the prompt.
  4. The obtained OAuth token is cached at `~/.config/bakku/gdrive-token.json`
     (mode `0600`) and automatically refreshed/reused on subsequent runs.

### Dropbox

```
dropbox://path
```

- `path` is a `/`-separated path under the app/account root.
- **Credentials**: set `BAKKU_DROPBOX_TOKEN` to a Dropbox API access token
  (long-lived or refreshable, as issued by your Dropbox app).
- Files up to 150MB use a single `Upload` call; larger files are
  automatically split into a Dropbox upload session (`UploadSessionStart` /
  `UploadSessionAppendV2` / `UploadSessionFinish`, 32MB chunks) since
  Dropbox's single-shot upload endpoint caps at 150MB per request.

### SMB / CIFS

```
smb://user@host/share/path
```

- `user` may be `DOMAIN\user` or `DOMAIN;user` to specify a domain/workgroup;
  otherwise no domain is sent.
- **Credentials**: password read from the `BAKKU_SMB_PASSWORD` environment
  variable; authentication uses NTLMv2.
- Saves are written to a temporary name on the share and renamed into place
  for a pseudo-atomic upload; intermediate directories are created as
  needed.

## Scheduling

`bakku schedule` installs a recurring job using the current OS's native
scheduler, so backups run without a long-lived bakku process or an external
cron daemon:

| OS | Backend |
|---|---|
| Linux | a `systemd --user` service + timer under `~/.config/systemd/user/`, enabled with `systemctl --user enable --now` |
| macOS | a `launchd` agent plist at `~/Library/LaunchAgents/com.bakku.<job>.plist`, loaded with `launchctl load -w` |
| Windows | a Scheduled Task created via `schtasks.exe` under the `Bakku\` folder |

```sh
# Install: run `bakku backup ~/Documents --repo laptop` every day at 03:00.
bakku schedule install --name daily-docs --cron "0 3 * * *" -- backup ~/Documents --repo laptop

# List every bakku-managed scheduled job on this system.
bakku schedule status

# Show a single job.
bakku schedule status --name daily-docs

# Remove it.
bakku schedule uninstall --name daily-docs
```

Notes:

- The cron expression is standard 5-field (`minute hour day-of-month month
  day-of-week`), validated with `github.com/robfig/cron/v3`'s standard
  parser before anything is installed.
- Everything after the literal `--` is the exact `bakku` subcommand and
  arguments to run; bakku does not interpret or rewrite it beyond shell
  quoting it into the generated unit/plist/task.
- **Linux**: `*/N` step and comma-separated list fields are expanded to
  explicit `OnCalendar=` value lists (systemd's calendar syntax has no
  direct `cron`-style step operator across all fields, so bakku expands the
  cron expression itself rather than emitting non-portable syntax).
- **macOS**: launchd's `StartCalendarInterval` supports only a single fixed
  value (or "every value") per field per dict, so a cron expression whose
  fields expand to multiple values (e.g. `*/15` minutes) is translated into
  the Cartesian product of matching dicts in the array — one dict per
  concrete (minute, hour, day, weekday, month) combination that should
  trigger a run.
- **Windows**: the `schtasks.exe` backend supports a fixed daily/weekly/
  monthly time (a single minute+hour value), matching the documented
  `"0 3 * * *"`-style use case; cron expressions combining
  day-of-month + day-of-week + month constraints, or minute/hour step/list
  fields, are rejected with a descriptive error rather than silently
  installing an incorrect schedule.
- If the underlying scheduler command (`systemctl`/`launchctl`/`schtasks`)
  fails or is not found in `PATH` (e.g. no systemd user session, no
  `launchctl` in a minimal environment), the generated unit/plist file is
  still written to disk and the exact manual command to finish registering
  it is printed as part of the error.

## Notifications

`backup` and `prune` can POST a webhook notification when they finish,
configured in the `[notify]` section of `config.toml`:

```toml
[notify]
webhook_url = "https://hooks.slack.com/services/T000/B000/XXXX"
on_success = true
on_failure = true
format = "slack"       # "slack", "discord", or "json" (default; auto-detected from the URL if omitted)
timeout_seconds = 10   # optional, default 10
```

The posted payload (as raw JSON for `format = "json"`, or as a one-line
summary embedded in `{"text": ...}` / `{"content": ...}` for
`slack`/`discord`) always carries:

```json
{
  "job": "backup",
  "status": "success",
  "hostname": "laptop",
  "snapshot_id": "a1b2c3d4...",
  "stats": { "files_new": 12, "chunks_new": 34, "...": "..." },
  "error": "",
  "duration_ns": 2500000000,
  "time": "2026-07-03T12:00:00Z"
}
```

- Notification delivery is always best-effort: a failed webhook POST (DNS
  failure, non-2xx response, timeout, etc.) prints a `warning:` line to
  stderr and never changes the exit status of `backup`/`prune`.
- `on_success` / `on_failure` are independent switches; set only one to
  only be notified about that outcome.
- Pass `--no-notify` on a single `backup`/`prune` invocation to skip sending
  a notification for that run without touching the config file.

## Design overview (repository format)

A bakku repository is a flat set of encrypted objects addressed by content
hash, stored under a single root (local directory or remote backend prefix):

```
config                 repository config (JSON)
keys/<id>              wrapped master key (argon2id + AES-GCM)
data/<xx>/<packID>     pack files (~16 MiB, many blobs + encrypted header)
index/<indexID>        encrypted blob -> pack location index
snapshots/<snapID>     encrypted snapshot records
```

Data flow during `backup`:

1. Each file is split into content-defined chunks by the keyed FastCDC
   chunker (`internal/chunker`), so identical byte runs across files/versions
   produce identical chunk boundaries.
2. Each chunk is content-addressed by its BLAKE3 hash (`internal/crypto`); if
   a blob with that hash is already present in the index, it is skipped
   (deduplication) — otherwise it is zstd-compressed, AES-256-GCM encrypted
   with a key derived from the repository's master key, and appended to the
   current in-progress pack (`internal/repo/pack.go`).
3. Packs are flushed to the backend at ~16 MiB, and the blob -> (pack, offset,
   length) mapping is recorded in an encrypted index (`internal/repo/index.go`).
4. A directory tree of the backed-up paths is built from file/dir/symlink
   nodes referencing content blob IDs (`internal/repo/tree.go`), encrypted, and
   stored as a snapshot record (`internal/repo/snapshot.go`) alongside
   hostname/paths/tags/timestamp metadata.

`restore` reverses this: read the snapshot record, walk its tree, resolve
each file's blob IDs via the index, fetch/decrypt/decompress the packs, and
write files back out (`internal/restorer`). `forget`/`prune`
(`internal/pruner`) compute which blobs are still reachable from any
remaining snapshot, then delete fully-unreferenced packs and repack
partially-referenced ones to reclaim space.

The repository password never touches disk or the wire: it derives an
argon2id key that unwraps a random master key stored (wrapped) under
`keys/<id>`; all other on-disk objects are encrypted with subkeys derived
from that master key.

## Not yet implemented (later phases)

- Volume Shadow Copy Service (VSS) on Windows: `--use-vss` validates the flag
  and prints a warning, but does not actually snapshot the volume; design
  notes for a full COM-based implementation are in
  `internal/fs/fs_windows.go`.
- Windows owner security-descriptor (SID) *restore*: the SID is read
  best-effort (`fs.OwnerSID`) but not yet written back on `restore`.
- Special files (block/char devices, FIFOs, sockets) are skipped during
  `backup` rather than archived.

## Testing

```sh
go test ./internal/...
```

Notable coverage added alongside OS metadata/scheduling/notifications:

- `internal/scheduler`: pure unit tests for cron-field expansion and the
  cron -> `OnCalendar=` / cron -> `StartCalendarInterval` converters (no OS
  scheduler binary required).
- `internal/notify`: `httptest`-backed tests verifying JSON/Slack/Discord
  payload shapes, format auto-detection, `on_success`/`on_failure` gating,
  and that webhook failures surface as non-fatal errors.
- `internal/fs`: ownership capture, hard-link (same inode) detection, and
  xattr round-trip/quarantine-exclusion tests (the latter two skip
  gracefully if the test filesystem does not support extended attributes).
- `internal/e2e`: a hard-link backup/restore test asserting a 3-way
  hard-linked file is chunked once and restored as a real hard link (shared
  inode) rather than duplicated content, in addition to the existing
  backup/restore and forget/prune/check/restore end-to-end tests.

## CI/CD

`.github/workflows/ci.yml` runs on every push/PR to `main` and on `v*` tags:

- **test** — `go vet ./...`, `go build ./...`, and `go test ./...` on
  `ubuntu-latest`, `macos-latest`, and `windows-latest`.
- **cross-build** — `CGO_ENABLED=0` builds of `./cmd/bakku` for all six
  release targets (darwin/linux/windows × amd64/arm64), uploaded as workflow
  artifacts on every run.
- **release** — on tags matching `v*`, runs `scripts/build-release.sh` and
  publishes the resulting `dist/bakku-*` binaries (plus a checksums file) to
  a GitHub Release for that tag.

To cut a release: tag `vX.Y.Z` and push the tag; the release job builds and
attaches binaries automatically. Locally, the same artifacts can be produced
with:

```sh
scripts/build-release.sh v1.2.3
```

which writes `dist/bakku-v1.2.3-<os>-<arch>[.exe]` for all six targets plus
`dist/bakku-v1.2.3-checksums.txt`, with the version/commit/date baked in via
`-ldflags` (see `bakku version`).
